# harness_eng

## What This Is

A controlled experiment that holds one LLM constant (Claude Sonnet) and varies the agent harness around it across five implementations, then measures the spread in task success rate, cost, and latency on a deterministically-graded HTML extraction benchmark. Ships as a runnable repo plus a drafted article written off the repo's own output.

## Core Value

Produce concrete, reproducible evidence — numbers and annotated failure traces — that harness design dominates model choice within a tier. If the repo runs and produces a chart showing large spread across harnesses on the same model, the project has succeeded.

### Out of Scope

- Multi-provider model support — single frozen Claude Sonnet is the experimental control; adding providers would contaminate the comparison
- Beyond 5 benchmark tasks — the pitch asked for 40, but the portfolio value is in the harness engineering, not fixture volume; user can extend the JSONL file later
- SWE-bench Lite or WebArena — chosen against because the grader would need docker/test execution and double the engineering surface without changing the story
- Parallel-run orchestration across machines — sequential is fine at 5×5 size
- Production observability (OpenTelemetry, W&B, MLflow) — JSONL traces are enough for the article
- Tuning the losing harnesses after seeing results — doing this invalidates the comparison; harnesses freeze before the full matrix runs

## Context

- **Origin**: proposed as a portfolio piece demonstrating "harness engineering" — the idea being that the current AI-eng discourse is model-obsessed and underprices scaffolding work.
- **Hypothesis to publish**: same model + different harness will produce a 2–4x spread in success rate and a 5–10x spread in cost. Publishing numbers close to that is the whole point.
- **Task choice**: structured extraction from messy HTML because it (a) has a trivial deterministic grader (normalized exact match), (b) needs no sandboxed code execution, (c) is runnable on a laptop, and (d) exposes harness differences — on clean HTML, single_shot always wins.
- **Traces from call 1**: deliberate. Retrofitting trace writers after a run produces nothing; the evidence for the "surprising failures" article section must exist before you know what surprised you.
- **Already built at time of project init**: pyproject.toml, .gitignore, .env.example, src/harness_eng/{config,model,trace,tools,grader}.py, src/harness_eng/tasks/{loader.py, tasks.jsonl, fixtures/*5}, harnesses/{base,single_shot,react,plan_execute}.py, README, .planning/ directory. Roadmap should structure the remaining work (two harnesses, runner, analysis, article generator, trace viewer, cost estimator, CI, tests) and retroactively validate the already-written scaffold.

## Constraints

- **Tech stack**: Python 3.11+; anthropic SDK as the only model client; beautifulsoup4+lxml for HTML; pandas+matplotlib for analysis; pytest for tests. Chosen for minimum-surface-area reproducibility.
- **Model**: claude-sonnet-4-6, temperature 0, max_tokens 2048, frozen in config.py. Any code path that touches anthropic outside model.py is a defect.
- **Budget**: API spend must be dry-run estimated before any full matrix execution. A 5-task × 5-harness matrix is small; a multi-seed full run on a frontier model can still surprise. Estimator is a gate, not an afterthought.
- **Timeline**: long-weekend scope. If any feature takes more than a few hours, it gets cut or simplified, not extended.
- **Harness freeze discipline**: once harnesses are implemented and reviewed, they do not get "fixed" after seeing results. Iterating on losing harnesses is a portfolio-killing methodology error.
- **Platform**: Windows dev machine; must run in bash under Git for Windows. No docker, no WSL requirement.
